Summary

Updated 3/10/19

I have been working at Electrolux for one year. I was hired as a Connectivity Intern with the responsibility of handling quality assurance for the Air Conditioners and Dehumidifiers. The main people in my team are my manager, the developer, and my x-coworker. He was an intern too and had been there for a year before leaving. My manager handled delegation of any tasks that needed doing. The developer would release updates, I would find bugs in them, and the results would be sent back to the developer. The app itself is the main focus of my job. The Frigidaire app can control the temperature, humidity, mode, and speed of an appliance from anywhere with an internet connection. In the beginning, I mostly did manual testing on the appliances with the mobile app, but as time went on I got more and more opportunities to develop software.

Work Completed

As I mentioned previously, my main goal as an intern is quality assurance for the Frigidaire App. Whenever there is an update, it’s my job to make sure there aren’t any bugs. Any bugs I find, I report on JIRA. JIRA is a bug tracking website, among other things. Every week I also file a report on the progress of registration of the week prior. I build charts and present the data to managers and my coworkers. In the beginning, I organized the data by hand, but now it is almost fully automated. I have also worked to support the voice command assistance offered by the Frigidaire mobile app. That means testing on the Alexa and Google Home devices.

Technologies and Skills



Git - I can easily say that my skillls in using Git have vastly increased since I started my job at Electrolux. I don’t really run into any problems with Git anymore, and I really appreciate being able to use Git so freely. Aside from version control, there are tons of other benefits to using Git. The way that I code, I will make a number of changes to my program. When I am done, I read through the “differences” in the Git GUI. This allows me to do my own code reviews, and I get an opportunity to go back and document the code. I also might notice something stupid I did or remember to remove some unused variables or functions. In short, I am way more organized with Git than I was without it. Although, this is not the intended purpose of Git. If I make any huge mistakes or delete something on accident, I can always revert to an old commit if I ever have the need. This gives me so much peace of mind since I don’t have to worry about slowing down to think super carefully before I remove something or rewrite it. I can test things out much faster this way.

Presentation - It didn’t seem like a big deal to me right away, but I have come to find that it makes a really huge difference if you know how to present your work well. I think the reason it’s so important to me is that no matter how hard of a worker you are or how much of a master at code you might be, if you can’t explain what you’re doing then you can’t convince people that you’re worth investing in.

Information Gathering - Another skill that seems like something trivial is information gathering. It’s super important to be able to sit down with your coworkers and really absorb everything they have to share with you. If you are scheduled to meet with someone and you can’t hold on to anything they say, then people will not want to work with you. That isn’t something that happened! I’m just saying that because I immediately recognized that my coworkers would make great resources and friends.

Appium and Automation Testing - The most obvious of all would be Appium. This is the tool I’ve spent the most time with. I can comfortably achieve anything I want in Appium now. I’m especially learned in Appium because of my experience with Hybrid apps. This means that I know how to deal with apps of any flavor. I’ve run into many obstacles, but I’ve managed to find my way every time. I know how to automate a span element. I know how to find text on the screen. I know how to click an element that isn’t available to locate in the native context, and can’t be clicked in the hybrid context. Aside from a select few bugs, I’ve learned a lot about code sleuthing through my Appium work. As a matter of fact, I’m suspicious that the bugs that are in my code might be originating from actual bugs in the app I’m meant to be testing, which means I’ve found bugs with automation software that were undetectable by manual testing.

CR1: Review 2/22/19

Here is my summary for the work I have completed this week:

There are three different appliances that are controller by the Frigidaire app: the Window Remote Air Conditioner (aka RAC), Portable Remote Air Conditioner (aka Stromboli), and Dehumidifier (aka Dehum). The air conditioners and dehumidifiers have similar visual control screens, but very different controls. However, the two AC units have much more similar controls. The only difference between the portable and window AC units is that the portable unit has a dry mode function. This week I worked on the automation for the Dehumidifier controls. There are two control options for the Dehumidifier. It has a speed setting and a humidity setting.

I set up the automation so that Appium, a mobile automation framework for Java, will store the values of the current humidity setting and speed setting. The test will tell the app to increase each respective setting and check to verify that they have changed to the correct values. If speed is set to low then increasing it should set it to high. Similar test conditions are fulfilled for the decrease functions. There weren't any problems with the speed automation. Those tests automated successfully and passed without any issues. There was an issue with the humidity automation. The lowest humidity setting is "Cont" which stands for "continuously running". The issue with this that the rest are based off numbers. There's something odd going on for "Cont" that I need to figure out since it's displaying as "cont" but the actual data value retrieved from the element is something different.... I'll need to debug this next week.

CR2: 3/1/19

This week I learned that talking to experienced developers is really important. There is always someone who knows more than you on any given subject. I talked to the App Developer this week and we did a code review of the project. A webview is a viewport used in Hybrid mobile applications as opposed to native views which are native to specific phones and OS (ios vs android). Frigidaire is a Hybrid App. There is a function in the code that will translate the elements' coordinates in a webview into coordinates that the native device can understand. Because the webview width/height do not match the native view width/height a ratio must be calculated. Even with these calculations, we have been seeing inaccuracy in the automations' ability to find vertical element location. This was odd considering the x location worked so well. Not only did x work, but the y worked perfectly on specific devices such as the Galaxy S7.

We reviewed the code and devices used in automation and discovered that the phones that have onscreen buttons (in the native space) were the exact height of the innaccuracy seen in our vertical offsets! We just need to find some way of returning the height of those buttons and that will likely fix our problem...

CR6: 4/5/2019

This week I worked on the AC automation. I was alter the code so that it is much more accurate in automating and verifying the functionality of the temperature. Since the range of the temperature goes from 60-90 fahrenheit, the app will loop back around to 60 if you try to increase the target temperature while you’re already at 90 degrees. Getting a human to understand this is pretty simple, but it was sort of complicated to put together in code, especially since there is an option between celsius and fahrenheit. I also made lots of changes to make the automation more viable for actual usage. Used junit to run parameterized tests so that I can run the same code in four different appliance modes. I also moved done code in the initialize method to increase efficiency.

CR9: 4/25/2019

CR10: 5/11/19

This is the final week of the project. I was able to complete everything I set out to accomplish. This week there was a lot to do, but because exams were cancelled I was able to dedicate the entire week to the project. I had support for RAC and STROMBOLI, and the tests for the Dehumidifier had already been written, but not parameterized. The Sign In tests also needed to be parameterized and there still was no GUI to run the program from.

The tasks for this week:

  1. Parameterize Dehumidifier Tests
  2. Parameterize Sign In Tests
  3. Design GUI
  4. Implement back end for GUI to control which tests are run
  5. Develop stress tests for co-worker
It took a long time to parameterize because I needed to be sure that none of the tests would negatively interact with each other. However, because the Dehumidifier and Sign In tests aren’t particularly complicated (in comparison with the Rac and Stromboli) this did not take very long to do. The GUI took some time to set up since I chose to design it in Netbeans then port it over to Eclipse. I’ve used both program’s swing designers and thought Netbeans was (although lacking in other areas) had a great UI designer built in. When I was done all I needed to was copy and paste the code. For the most part the UI just lets you select which appliance you want to test and then hit the GO button. In the future the UI could include options for specific testing on that appliance like “Temperature Only” options. I made sure to include logic to disable whichever tests I didn’t running in combination with each other, as well as including an option for which phone the test would run on.

For the backend of the UI I chose XML to port the configurations over between the GUI and the JUNIT class because I wasn’t sure how the JUNIT class work the new set up. Looking back, I now realize the XML probably wasn’t necessary, but now I’ve learned how to read/write XML so that’s not really a bad thing. Once the XML is read by the JUNIT class, the tests will run (or not run) depending on the configurations.

My co-worker mentioned how useful it would be to able to run repetitive testing throughout the day, so the final stage was adding functionality to my program to support stress testing configurations. All I really needed to do was add support to the GUI and add a condition that would repeat all tests 10x over. The dehumidifier test was able to run 10 tests within 13 minutes! That’s a major accomplishment to me. When I started each tap would take up to 10 seconds and I have reduced that to less than 1 second.
